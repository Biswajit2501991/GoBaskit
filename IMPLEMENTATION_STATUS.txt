================================================================
1) WHAT HAS BEEN IMPLEMENTED SO FAR
================================================================

Step 1 - Mobile Number Detection (Implemented)
- Admin Login is hidden by default for normal customers.
- Header uses "My Account" modal to check mobile first.
- API checks staff existence in DB table `staff_accounts`.
- "Login as Admin" button appears only for staff mobile detection.
- Detection is session-memory based (in-memory Zustand store), so refresh resets it.

Step 2 - Admin Login + Secure Session (Implemented)
- Mobile+password staff login modal is implemented.
- Passwords are bcrypt hashed and compared via hash only.
- Access token + refresh token flow implemented:
  - Access JWT cookie
  - Refresh token cookie (`staff_refresh_tokens`)
- Remember Me supported by longer refresh token TTL.
- Logout supported and clears auth cookies.
- Failed login tracking + rate limiting implemented (`staff_login_attempts`).
- Audit logging for login success/failure is implemented (`audit_logs`).
- Legacy email admin login remains backward compatible.

Step 3 - Staff Management (Implemented, core)
- Staff CRUD APIs and UI implemented.
- Staff list/search and activation/deactivation support implemented.
- Roles + permission model implemented in code.
- Permission-gated navigation and server-side route protection added.

Step 4 - Bulk Staff Import (Implemented, core)
- Staff import template download endpoints implemented.
- Preview + validation implemented:
  - duplicate mobile
  - duplicate email
  - invalid mobile
  - invalid role
- Transactional import with rollback behavior via Prisma transaction.
- Import report output implemented.

Step 5 - Admin Dashboard (Partially Implemented, high-value)
- Dashboard upgraded from legacy stats to richer KPI cards.
- Cached dashboard service implemented with invalidation on key events.
- Includes orders/revenue/staff online/low stock/recent activity/top products/trends.

Step 6 - Product Management (Partially Implemented)

- Staff permission checks added to product/category pages and APIs.

- Full advanced product CMS features (drag-drop ordering, multi-image gallery, SEO, restore, etc.) still pending.

Step 7 - Home Page Configuration (Partially Implemented)

- Customer homepage now consumes this config via `/api/config`.
- Full section-level CMS (offers/coupons/seasonal/footer/header block builder) still pending.

Step 8 - Order Management (Implemented foundation)
- Orders API and manager upgraded for staff-secure workflows.
- Pagination, search/filter, optimistic updates implemented.

Step 9 - Order Assignment + Locking (Implemented foundation)

================================================================
2) IMPORTANT DATABASE/ARCHITECTURE DETAILS
================================================================

Core new/extended DB entities:
- staff_accounts
- staff_refresh_tokens
- staff_login_attempts
- audit_logs
- admin_notifications
- order_status_history
- orders extended with:
  - assigned_staff_id
  - locked_at
  - priority
  - admin_notes

Key architecture decisions:
- Reused existing `/admin/*` structure (no duplicate admin app).
- Kept legacy admin auth compatibility while introducing staff-first auth.
- Real-time admin events use SSE event bus (`/api/admin/events`).
- Config and KPI reads use cache-backed services to reduce DB traffic.

================================================================
5) OPERATIONAL NOTES FOR NEXT AGENT/DEVELOPER
================================================================

- Always use `requireStaffPermission(...)` / `requireStaffSession()` for new admin routes.
- Keep customer flow unchanged when extending admin features.
- Avoid duplicate APIs; extend existing services/modules first.
- Invalidate relevant caches when mutating related entities:
  - settings changes -> SettingsService cache
  - order/status/notification changes -> Dashboard/Analytics caches
- Keep optimistic UI updates paired with robust server-side revalidation fallback.
- Preserve backward compatibility with legacy `/api/auth/login` until explicitly removed.
